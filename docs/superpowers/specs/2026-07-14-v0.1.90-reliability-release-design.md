# Codex Remote Bridge v0.1.90 Reliability Release Design

## Objective

Ship one coordinated Web and Windows desktop release that reduces remote RPC load, removes
internal Codex screenshots from the public conversation, fixes Windows path and timeout failures,
keeps desktop branding stable after upgrades, and gives the desktop agent a visible update check
with an administrator-elevated installer. It also replaces large inline plan cards with the
approved compact composer progress popover.

## Current Problems

1. `getPathParent('K:\\codex-mcp')` returns `K:`. Windows treats that as drive-relative, so the
   desktop local operation rejects it and `/codex-api/project-root-suggestion` returns 502.
2. project-name suggestions run during startup, focus, folder-list watchers, and unrelated project
   actions. The operation is only needed while opening the create-project flow.
3. `thread/read` and `thread/resume` return the complete thread from Codex Desktop to the relay.
   The public server trims the result afterward, so large sessions still consume Agent memory,
   WebSocket bandwidth, and most of the 45-second relay timeout.
4. notification compensation forces `thread/list` for broad `thread/*` and `turn/completed`
   events. It does not read every thread body, but repeatedly reloads the first 50 summaries.
5. Codex `imageView` tool results are normalized as assistant-generated images. Local diagnostic
   screenshots therefore appear as broken images on the public Linux server.
6. Windows may retain the old taskbar icon for an upgraded executable at the same install path.
7. the NSIS package installs for the current user and the agent has no visible release check.

## Web And Relay Design

### Selective thread synchronization

Notifications remain the source of truth. A small policy helper classifies each event as one of:

- update local state only;
- mark one `threadId` dirty;
- refresh thread-list membership;
- recover all state after reconnect.

Title, status, token, turn, and session-file events must not force `thread/list`. The active dirty
thread is read after the existing 220 ms debounce. An inactive dirty thread remains in a set and is
read only when selected. `thread/list` is reserved for initial/reconnect recovery and membership
changes such as creation, archival, deletion, and explicit refresh.

Realtime item deltas continue to render immediately. A final targeted read reconciles persisted
turn data after completion without reloading unrelated summaries.

### Desktop-side response trimming

The CDP renderer adapter trims `thread/read`, `thread/resume`, `thread/fork`, and
`thread/rollback` results to the latest 10 turns before returning the value to Rust. It adds
`threadTurnStartIndex` so older-turn pagination remains correct. The relay timeout increases from
45 to 90 seconds as a guard, not as the primary performance fix.

### Project root behavior

Windows drive roots are preserved as `K:\\`. Project-name suggestions run only when the create
project modal is opened or after a project operation that leaves that modal open. Startup, focus,
folder-option watchers, and unrelated imports do not request a suggestion.

### Image visibility

`imageView` is an internal Codex tool-display item and is excluded from both historical
normalization and realtime `item/completed` handling. User-uploaded message images and explicit
`imageGeneration` results remain visible. No local image files are deleted.

## Composer Plan Progress

The Web conversation excludes `plan` and `plan.live` messages from inline rendering. A shared plan
selector finds only the latest incomplete plan for the selected thread and computes the current
step from the in-progress step or next pending step. `PlanProgressPopover` renders directly above
the composer, opens upward on hover/focus/tap, and disappears as soon as every step completes.
Completed historical plans remain hidden after reload. User, assistant, command, file, and system
messages retain their current ordering and rendering.

The component uses a real button, progress ARIA values, text/icon status distinctions, wrapped
mobile step labels, and a popover that does not change composer height.

## Desktop Agent Design

### Update check

The agent queries `https://api.github.com/repos/YusenDu/codex-remote-bridge/releases/latest` with a
bounded timeout and explicit User-Agent. It compares the release tag with the running package
version and returns a typed status to the WebView.

The settings view contains a manual **Check for updates** action and shows current/latest version,
network failure, up-to-date, or update-available state. One background check runs after startup.
When an update exists, the action opens the exact GitHub Release page in the system browser. This
release does not silently replace binaries because unsigned unattended replacement would weaken
the Windows trust boundary.

### Administrator installation and icon refresh

NSIS `installMode` changes to `perMachine`, which installs under Program Files and requests UAC
administrator approval. An `NSIS_HOOK_POSTINSTALL` hook runs `ie4uinit.exe -ClearIconCache` and
`ie4uinit.exe -show` when available. It does not restart Explorer. Existing unified ICO/PNG assets
remain the sole source for the executable, installer, uninstaller, tray, taskbar, and in-app logo.

## Version And Publishing

Web, desktop package, Rust crate, Tauri bundle, preview UI, source archive, prebuilt archive,
portable executable, and NSIS installer use `0.1.90`. A consistency test prevents version drift.

After verification, copy the reviewed source changes into the `codex-remote-bridge-publish`
checkout, commit and push to `YusenDu/codex-remote-bridge`, then publish the four release artifacts
and SHA-256 manifest under tag `v0.1.90`.

## Verification

- Unit tests prove Windows drive-root preservation, selective event policy, inactive dirty-thread
  deferral, `imageView` suppression, `imageGeneration` preservation, renderer-side turn trimming,
  relay timeout, latest-incomplete-plan selection, inline plan suppression, popover interaction,
  update parsing/comparison, NSIS elevation/hook configuration, and version parity.
- Full Vitest, Vue/CLI build, Rust tests, release Rust build, and NSIS packaging must pass.
- Browser QA verifies user images remain, tool screenshots disappear, active-thread changes sync,
  inactive-thread changes do not trigger full reads, stop state works, and no repeated list reloads
  occur for ordinary turn events.
- Windows QA verifies the settings update state, Release-page action, UAC manifest behavior,
  installed/tray/taskbar/in-app icons, startup behavior, and QR/device routing.

## Non-goals

- Silent self-update or background binary replacement.
- Deleting Codex screenshots from the desktop filesystem.
- Replacing the event stream with periodic full polling.
- Deploying the server from this release task; publishing GitHub source and artifacts is required.
